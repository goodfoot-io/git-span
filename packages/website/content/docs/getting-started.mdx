## Install

```bash
npm install -g git-span
```

Verify the install:

```bash
git span --version
```

```txt
git-span 1.0.134
```

## Set up a toy repo

Create a repo with two files that are coupled but not enforced by anything: an API that validates a minimum charge amount, and a checkout UI that duplicates the same minimum.

```bash
mkdir git-span-tutorial && cd git-span-tutorial
git init
mkdir -p api web
```

```bash
cat > api/charge.ts <<'EOF'
export const MIN_AMOUNT_CENTS = 50;

export function charge(amountCents: number) {
  if (amountCents < MIN_AMOUNT_CENTS) {
    throw new Error("amount too small");
  }
  return amountCents;
}
EOF
```

```bash
cat > web/checkout.tsx <<'EOF'
export function Checkout({ amountCents }: { amountCents: number }) {
  if (amountCents < 50) {
    throw new Error("amount too small");
  }
  return amountCents;
}
EOF
```

Commit the starting point:

```bash
git add api web
git commit -m "Add charge API and checkout UI"
```

Nothing here says these two files must agree on `50`. That's the coupling a span declares.

## Declare a span

`git span add` takes a span name and one or more anchors. An anchor is either a whole file or a `path#Lstart-Lend` line range (1-indexed, inclusive). Anchor the validation block in `api/charge.ts` by line range, and anchor `web/checkout.tsx` as a whole file:

```bash
git span add checkout/min-amount api/charge.ts#L4-L6 web/checkout.tsx
```

`api/charge.ts#L4-L6` covers the `if`/`throw`/closing brace that checks `MIN_AMOUNT_CENTS`.

## Attach a why

A span without a why is just a list of locations. Write one sentence defining what the anchors collectively do:

```bash
git span why checkout/min-amount -m "Minimum-amount validation that the charge API and the checkout UI must agree on."
```

## Commit the span

Span data lives under `.span/`, tracked by git like any other file:

```bash
git add .span && git commit -m "Declare checkout/min-amount span"
```

## Inspect the span

```bash
git span show checkout/min-amount
```

This renders the span as a TOML-style view: its `name`, its `message` (the why you just wrote), one `[[anchors]]` block per anchor (`id`, `path`, `extent`), and a `[config]` block (`copy_detection`, `ignore_whitespace`, `follow_moves`). That rendered view is a display format — the file actually stored under `.span/` uses a different, internal layout. See [Concepts](/docs/concepts) for the distinction.

<Callout type="info">
`git span list` shows the same anchors in a shorter form, and lists every span in the repo when you omit a name.
</Callout>

## Cause drift and catch it

Edit `web/checkout.tsx` so it no longer agrees with `api/charge.ts`, without touching the anchored file on the other side:

```bash
cat > web/checkout.tsx <<'EOF'
export function Checkout({ amountCents }: { amountCents: number }) {
  if (amountCents < 100) {
    throw new Error("amount too small");
  }
  return amountCents;
}
EOF
```

Now scan for drift:

```bash
git span stale
```

```txt
## checkout/min-amount
- api/charge.ts#L4-L6
- web/checkout.tsx — changed in the working tree

Minimum-amount validation that the charge API and the checkout UI must agree on.
```

```bash
echo $?
```

```txt
1
```

The listing shows every anchor in the stale span, not just the drifted one — `api/charge.ts#L4-L6` is printed with no suffix because it hasn't moved or changed — followed by the span's why, so the report itself reminds you what the coupling is for. `git span stale` exits `1` whenever it finds drift; on a clean scan it prints a `0 stale across N spans` summary and exits `0`. Once you decide whether the drift was intentional and re-anchor with `git span add` (or fix the underlying disagreement in code), a rerun of `git span stale` reports `0 stale` and exits `0`.
