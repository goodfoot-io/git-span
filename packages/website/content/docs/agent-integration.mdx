The git-span plugin is how an agent operates git-span without being told the command syntax on every turn. Once installed (see [Getting started](/docs/getting-started)), it ships four things:

- **A touch hook** ‚Äî fires after every file read/edit/write, healing positional drift inline.
- **A commit gate** ‚Äî fires before `git commit`/`git push`, holding the command when the changeset carries real span debt, and advises (without holding) on a plain `git status`.
- **Two skills** ‚Äî `git-span` (command syntax, recipes, gate-denial handling) and `reconcile` (the multi-span triage workflow), loaded into context when the agent's task matches.
- **An expert subagent** ‚Äî a dedicated agent for span work: creating, reconciling, and inspecting spans, with judgment about when a coupling is real and when a span should be deleted rather than patched.

The CLI itself doesn't change: every capability the plugin uses is a plain `git span` invocation, scriptable the same way in a terminal or CI.

## Touch hook + gate

- **Touch hook** (`PostToolUse`) ‚Äî fires synchronously right after each file read/edit/write. It re-anchors pure positional drift (a line-shift from the edit) against the file's real post-edit content, silently ‚Äî no output, nothing in the transcript, because there's nothing left to act on. If genuine semantic drift survives that heal (the anchored content no longer matches what a span asserts), it injects a bounded context block naming the span, its status, and what to do. Each `(span, status)` pair surfaces at most once per session.
- **Gate** (`PreToolUse` on the shell tool) ‚Äî fires before `git commit`/`git push`/`git status` runs. It resolves the actual changeset (staged files, plus tracked-modified files under `-a`/`-am`; for `git status`, staged plus tracked-modified, the same working-tree picture `git status` itself prints), reruns a scoped `git span stale --fix`, and classifies what's left. For `git commit`/`git push`, real span debt ‚Äî semantic staleness, or a changed file no span covers ‚Äî denies the command with a checklist once per distinct debt state; an identical retry proceeds. `git status` never denies: the same checklist surfaces as advisory context instead, re-reported live on every call since a status preview has no memory of what it already showed. Positional drift never blocks; the touch hook already healed it. If the scan itself fails to complete, the gate never blocks on that account ‚Äî it warns that span debt wasn't verified and lets the command proceed.

<Callout type="info">
Both hooks fail open at every layer: a missing `git span` binary, a timeout, or an unexpected result resolves to "allow silently, inject nothing." Neither can brick an edit or a commit on its own failure.
</Callout>

### What the gate denies

Two shapes, both rendered into the denied command's `permissionDecisionReason`:

**Semantic staleness** ‚Äî the full span in the same human format `git span list` prints (all anchors, drifted ones labeled, the description), denied once per distinct set of findings; a retry with the same findings proceeds, and editing a span's anchors changes the findings and produces one fresh deny:

```
This change leaves an implicit dependency out of date:

## checkout-flow
- web/checkout.tsx#L4-L6 ‚Äî changed
- api/charge.ts#L30-L76

Checkout request flow that carries a charge attempt from the browser to the
Stripe-backed server.

---

Update the drifted locations or the description ‚Äî `git span add checkout-flow
<path#Lstart-Lend>` / `git span why checkout-flow -m "..."` ‚Äî then retry. If a
dependency no longer holds, tell the user instead.
```

**Uncovered writes** ‚Äî a changed file no span anchors at all, denied once per distinct debt state; an unchanged retry passes:

```
Decide whether these changed files carry an implicit dependency ‚Äî code
kept consistent with other locations that nothing links to it:
  - src/new-module.ts

If one exists: `git span add <name> <path#Lstart-Lend>` then `git span why
<name> -m "one sentence: the subsystem, what it does across locations"`.
Otherwise retry the command to proceed (one-time check).
```

`MOVED` and `RESOLVED_PENDING_COMMIT` anchors are never debt ‚Äî they never appear in either checklist and never block. `.span/**` writes are excluded from the uncovered-writes check so a span repair riding the same commit never self-triggers the gate.

A `git status` preview renders the same two checklists as advisory `systemMessage` context, minus the retry-flavored closing sentence (`‚Äî then retry`, `Otherwise retry the command to proceed`) ‚Äî nothing was ever held, so there's nothing to retry. Critically, a `git status` call never consults or updates the once-per-debt-state memo that `git commit`/`git push` rely on: it can't spend the one-time deny a later real commit depends on, and it always reports whatever debt is live right now.

### Suppression

There's no override for a gate deny ‚Äî every deny path resolves on its own: a fresh finding denies once, and an identical retry proceeds. What's left to configure is what the hooks pay attention to in the first place:

- `.span/.hookignore` ‚Äî a gitignore-like, per-path file suppressing the touch hook's context injection for noisy spans (wiki/marketing prose, etc). Never affects `git span` command output itself.
- `.span/.gateignore` ‚Äî user-owned; each non-comment line is a gitignore-style path pattern (same grammar as `.hookignore`, minus the span-slug-prefix list) excluding matching paths from the gate's uncovered-writes check. Nothing auto-creates it, and a missing/unreadable file fails open to no additional exclusion. It never suppresses the semantic-staleness check.

### Resolving a denied commit

1. Semantic staleness: fix each listed span (`git span add` the drifted anchors, or `git span delete` if the coupling is gone), then retry the same commit ‚Äî or just retry with the findings unchanged, since an identical set of findings only denies once.
2. Uncovered writes: either declare the coupling (`git span add` then `git span why -m "..."`) or just retry ‚Äî a second attempt at an unchanged debt state passes.

See [Reconcile stale spans](/docs/guides/reconcile-stale-spans) for the full triage workflow.

## Codex specifics

The trust flow in [Getting started](/docs/getting-started) is what actually turns these hooks on under Codex ‚Äî installing the plugin alone leaves them inert until `/hooks` review. Two things worth repeating here because they change what "installed" means:

- Hash-stable hook filenames keep an existing trust decision valid across plugin upgrades, so a Codex user doesn't re-trust on every update.
- Hooks are disabled entirely on Windows, with no partial fallback ‚Äî none of the touch-hook or gate behavior above applies there.

None of this affects Claude Code, which has no separate trust step.

## The enforcement backstop

The touch hook and gate only run inside a hooked agent session. A human-authored commit, a session with hooks disabled, or a Codex `permissionDecision: 'deny'` that doesn't actually block can all still land span debt. [CI & git integration](/docs/ci) covers `git span stale` as the backstop that catches what the in-session hooks miss.
