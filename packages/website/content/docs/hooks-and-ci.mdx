## Agent runtime: touch hook + gate

git-span ships as a plugin for both Claude Code and OpenAI Codex, wiring two in-session hooks that run without waiting for a commit:

- **Touch hook** (`PostToolUse`) — fires synchronously right after each file read/edit/write. It re-anchors pure positional drift (a line-shift from the edit) against the file's real post-edit content, silently — no output, nothing in the transcript, because there's nothing left to act on. If genuine semantic drift survives that heal (the anchored content no longer matches what a span asserts), it injects a bounded context block naming the span, its status, and what to do. Each `(span, status)` pair surfaces at most once per session.
- **Gate** (`PreToolUse` on the shell tool) — fires before `git commit`/`git push` runs. It resolves the actual changeset (staged files, plus tracked-modified files under `-a`/`-am`), reruns a scoped `git span stale --fix`, and classifies what's left. Real span debt — semantic staleness, or a changed file no span covers — holds the command with a checklist. Positional drift never blocks; the touch hook already healed it.

<Callout type="info">
Both hooks fail open at every layer: a missing `git span` binary, a timeout, or an unexpected result resolves to "allow silently, inject nothing." Neither can brick an edit or a commit on its own failure.
</Callout>

### What the gate denies

Two shapes, both rendered into the denied command's `permissionDecisionReason`:

**Semantic staleness** — hard-until-resolved; the same command re-denies on every retry until the findings themselves change:

```
This changeset carries span debt — resolve it before this lands:
  - checkout-flow (CHANGED): web/checkout.tsx#L4-L6

Update each span's anchors/why in this same change, or tell the user why the
described coupling no longer holds, then retry.
```

**Uncovered writes** — a changed file no span anchors at all, denied once per distinct debt state; an unchanged retry passes:

```
These changed files are covered by no span — consider whether they need one:
  - src/new-module.ts

Declare a coupling with `git span add` if one genuinely exists, or just retry
the command to proceed (this is a one-time check).
```

`MOVED` and `RESOLVED_PENDING_COMMIT` anchors are never debt — they never appear in either checklist and never block. `.span/**` writes are excluded from the uncovered-writes check so a span repair riding the same commit never self-triggers the gate.

### Escape hatch and suppression

- `GIT_SPAN_GATE=skip` — prefix the git command with this env var to bypass the gate for one invocation. It requires explicit user approval and is transcript-visible (a notice is emitted), never silent.
- `.span/.hookignore` — a gitignore-like, per-path file suppressing the touch hook's context injection for noisy spans (wiki/marketing prose, etc). Never affects `git span` command output itself.
- `.span/.gateignore` — user-owned; each non-comment line is a gitignore-style path pattern (same grammar as `.hookignore`, minus the span-slug-prefix list) excluding matching paths from the gate's uncovered-writes check. Nothing auto-creates it, and a missing/unreadable file fails open to no additional exclusion. It never suppresses the semantic-staleness check.

### Resolving a denied commit

1. Semantic staleness: fix each listed span (`git span add` the drifted anchors, or `git span delete` if the coupling is gone), then retry the same commit.
2. Uncovered writes: declare the coupling with `git span add`, or just retry — a second attempt at an unchanged debt state passes.

See [Reconcile stale spans](/docs/guides/reconcile-stale-spans) for the full triage workflow.

## CI gate: the enforcement backstop

The in-session gate only runs inside a hooked agent session — a human-authored commit, a session with hooks disabled, or (on Codex) a `permissionDecision: 'deny'` that doesn't actually block can all still land span debt. `git span stale`'s default exit code is the backstop, checked before merge:

```bash
git span stale
```

`git span stale` exits 1 when it finds drift and 0 on a clean scan, so it drops into any CI pipeline as a plain step:

```yaml
- uses: actions/checkout@v4
  with:
    fetch-depth: 0 # spans compare against real git history
- run: npm install -g git-span
- run: git span stale
```

For any other CI system:

```bash
git span stale || { echo "stale spans found — run 'git span stale' locally to see them"; exit 1; }
```

For a report-only job that should never fail the build, add `--no-exit-code`:

```bash
git span stale --no-exit-code
```

<Callout type="warn">
`--no-exit-code` only suppresses the exit code for ordinary drift. An interior-anchor integrity violation still exits 1 even with `--no-exit-code` set.
</Callout>

For machine consumption, use `--format json` or `--format porcelain`:

```bash
git span stale --format json
git span stale --format porcelain
```

<Callout type="info">
Both machine formats print nothing at all to stdout on a clean scan — no `{"findings": []}`, no `# porcelain v2` header. Empty stdout plus exit 0 is the clean signal; a parsing step that expects a JSON envelope on every run will break on the clean case. The human format (the default) prints a summary line on a clean scan.
</Callout>

See [Command reference](/docs/commands) for the full set of `stale` flags, and [Concepts](/docs/concepts) for what counts as drift.

### Wiring into a validate script

Add `git span stale` as its own step alongside lint/typecheck/test, run **after** them — span debt is a documentation-coupling concern, not a correctness one, so it should never mask an earlier, more actionable failure:

```bash
echo "Checking spans..."
git span stale || { echo "git span stale found drift — see above"; exit 1; }
```

## Local pre-commit hook

For a workflow without agent hooks at all (a human editor, or a harness git-span doesn't yet plug into), add a plain `git span stale` call to `.git/hooks/pre-commit`:

```bash
#!/bin/sh
git span stale
```

```bash
chmod +x .git/hooks/pre-commit
```

With no drift, the hook exits 0 and the commit proceeds. With drift, `git span stale` exits 1, which fails the hook and blocks the commit — fail-closed by default. A developer can still bypass it with `git commit --no-verify`, which is standard git behavior for any hook, not something git-span adds or removes.

## Merge driver

Registering `git span merge-driver` lets `.span/` files merge structurally instead of falling back to a textual diff3 merge. Add the attribute:

```txt
.span/** merge=span
```

And the driver in `.git/config`:

```ini
[merge "span"]
	name = git-span structural span merge driver
	driver = git span merge-driver %O %A %B %L
```

The driver resolves whatever is structurally derivable from the three sides without trusting the worktree. Genuine same-anchor divergence — both sides changed the same anchor's range or hash, or the `why` text diverged with no common merge base — it can't resolve on its own; it writes minimal conflict markers into the span file and exits non-zero so that a later `git span stale --fix` can finish the resolution. Anything still unresolved surfaces in `git span stale` with a `— conflict` suffix in human output (`CONFLICT` in the machine formats).

See the [reconcile stale spans guide](/docs/guides/reconcile-stale-spans) for what to do with a span left in that state.
